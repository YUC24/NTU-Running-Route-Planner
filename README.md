# [台大附近路跑路線規劃與揪團(暫)]

## Proposal Report

### 動機與目標
<!-- 說明為什麼想做這個專題 -->
隨著健康意識的提升，越來越多學生開始進行跑步運動。然而，在台大附近雖然有許多適合跑步的地點，例如校園周邊、公館商圈與河濱等，但對於初學者來說，常常會遇到不知道該跑哪裡、路線長度如何、或缺乏跑步夥伴等問題。

因此，本專題的目標是設計一個以台大附近為範圍的路跑路線規劃與揪團平台，讓使用者可以透過地圖查看跑步地點，規劃適合自己的路線，並建立或加入跑步活動，提升運動的便利性與互動性。

### 競品比較
目前市面上已有許多與運動、地圖或跑步相關的應用程式，例如 Google Maps、Strava、Nike Run Club 等。這些平台各自有不同的功能，但多數並不是專門針對「台大附近學生跑步」這個情境設計。

Google Maps 可以提供完整的地圖與路線資訊，但主要用途是一般導航，較少針對跑步者提供校園周邊跑步路線推薦，也沒有活動揪團功能。

Strava 可以記錄跑步數據，也能看到其他使用者的跑步紀錄，但對初學者來說，若只是想快速找到台大附近適合的跑步路線，操作上可能比較複雜，也不一定能直接建立小型校園跑步活動。

Nike Run Club 提供跑步紀錄與訓練計畫，但較偏向個人訓練與跑步紀錄，對於特定地區的路線規劃與校園揪團功能支援較少。

因此，本專題的差異化在於聚焦「台大附近」與「學生跑步揪團」兩個情境。系統不追求完整取代大型運動 App，而是希望提供一個簡單、直覺、適合初學者使用的跑步路線規劃與活動加入平台。

### 預期功能
<!-- 列出預計實作的功能 -->
1. 地圖顯示功能
   系統使用 Google Maps 作為地圖工具，在網頁上顯示台大附近的跑步地點，例如台大校園、公館、河濱等區域。
   這些地點會以 marker 的方式呈現在地圖上，讓使用者可以直觀看到各地點的位置。
2. 路線規劃功能：
   使用者可以選擇起點與終點，系統會根據目前設定的地點與路線資料，利用 Graph 結構與 Dijkstra 演算法計算出推薦路線。
   路線會顯示在 Google Maps 上，並回傳總距離與預估跑步時間，幫助使用者判斷該路線是否符合自己的體力與需求。
3. 活動建立功能：
    使用者可建立跑步活動（時間、地點、人數等）
4. 活動瀏覽與加入功能：
    顯示目前所有活動
    若活動尚未額滿，使用者可以加入活動；若已經加入，也可以退出活動。若活動人數已滿，系統會顯示「已滿」狀態，避免超過人數限制。

### Prototype 預計可驗證內容
本階段 Prototype 主要用來驗證系統的核心功能是否可行，包含地圖顯示、路線計算與活動功能。

首先，驗證 Google Maps 是否能正常顯示台大附近的地圖，並在地圖上標示跑步地點。

其次，驗證使用者選擇起點與終點後，系統是否能透過 Graph、Adjacency List 與 Dijkstra Algorithm 計算出一條路線，並顯示總距離與預估時間。

此外，也會驗證活動功能是否能正常運作，包括建立活動、顯示活動列表、加入活動、退出活動與刪除活動。

最後，Prototype 也會檢查前後端 API 是否能正確連接，讓前端可以成功取得地點、路線與活動資料。

## Prototype Report

### 目前進度
<!-- 完成了什麼 -->
目前專題已完成基本的系統架構規劃，並確定使用 Google Maps 作為地圖顯示工具。前端部分已建立基本頁面與地圖介面，能夠在地圖上顯示台大附近的地點 marker。

在路線規劃方面，目前已設計 locations 與 routes 資料，並以 Graph 與 Adjacency List 的方式表示地點之間的連線關係。後端已使用 Node.js 與 Express 建立 API，讓前端可以向後端請求地點資料與路線計算結果。

活動功能方面，目前已實作活動列表、建立活動、加入活動、退出活動與刪除活動的基本邏輯。活動資料以 JSON 檔案儲存，並透過 API 與前端互動。

### 遇到的困難
<!-- 遇到什麼問題、如何解決或打算如何解決 -->
目前遇到的主要困難是「路線計算結果與預期有明顯落差」。雖然系統已經使用 Graph、Adjacency List 與 Dijkstra 演算法來計算起點到終點之間的最短路徑，但實際算出的路線有時候會和預期中的跑步路線差很多。例如，系統可能會選到距離較短但不適合跑步，或是路線方向看起來不符合一般使用者的直覺。

造成這個問題的原因可能是目前 `routes` 資料中的節點與邊仍不夠完整，某些地點之間的連線或距離權重設定不夠準確，導致 Dijkstra 演算法雖然有正確計算最短路徑，但計算出的結果不一定是適合跑步的路線。另外，Dijkstra 演算法本身只會根據權重選擇路徑，不會判斷路線是否安全、是否適合跑步、是否需要頻繁過馬路，或是否符合使用者實際習慣。

目前的解決方向是重新檢查 `locations` 與 `routes` 資料，確認每個地點座標、節點連線與距離權重是否合理。接著會調整 Graph 中的邊與權重，讓演算法計算出的結果更接近實際適合跑步的路線，而不只是單純距離最短。

另一個困難是 Dijkstra 計算出的路徑要正確顯示在 Google Maps 上。後端回傳的是節點順序，但前端需要將這些節點轉換成座標並畫在地圖上。如果節點順序、座標資料或前後端資料格式不一致，就可能造成地圖上的路線顯示與預期不同。接下來會確認 API 回傳格式，並檢查前端是否正確依照回傳的節點順序繪製路線。

### 下一步計畫
<!-- 接下來要做什麼 -->
接下來會優先修正路線計算結果與預期落差過大的問題。首先會重新檢查 locations 與 routes 資料，確認每個地點的座標、節點連線與距離權重是否合理。接著會調整 Graph 中的邊與權重，讓 Dijkstra 演算法計算出的路線更接近實際適合跑步的路線，而不只是單純距離最短。

在路線顯示方面，會繼續修正 Google Maps 上的路線呈現方式，確認後端回傳的路徑順序能正確對應到地圖上的座標，並讓使用者可以清楚看到推薦路線、總距離與預估時間。

活動功能方面，會繼續優化活動卡片與按鈕樣式，使建立、加入、退出、已滿與刪除等狀態更清楚，也會測試不同使用者狀態下的操作是否正確。

最後，會進行整體系統測試，確認地圖顯示、路線規劃、活動建立、加入、退出與刪除功能都能正常運作，並整理最終報告與展示內容。

### 與課程的關聯
<!-- 到目前為止，你的實作中哪些部分與課程內容有關？關係是什麼？ -->
本專題目前最主要與課程中的 Graph、Adjacency List、Dijkstra Algorithm 以及 Array / List 有關。

在路線規劃功能中，系統將台大附近的跑步地點視為 Graph 的節點，地點之間的道路視為邊，並使用 Adjacency List 儲存各地點之間的連線關係。

當使用者選擇起點與終點後，系統會利用 Dijkstra Algorithm 根據路線距離計算最短路徑，這部分直接應用了課程中圖論與最短路徑的概念。

此外，活動功能使用 Array / List 管理活動資料與參加者名單，並進行新增、刪除、查詢與更新等操作，因此也和課程中的基本資料結構應用有關。
